# eboot Configuration Schemas

Machine-readable boot configuration contracts consumed by ebuild.
ebuild generates these configs; eboot consumes them deterministically.

## Files

| Schema | Purpose | Generated By | Consumed By |
|---|---|---|---|
| `boot.schema.yaml` | Boot config format — layout, policy, image, RTOS params | ebuild `EosConfigGenerator` | eboot `generate_config.py` |
| `flash_layout.schema.yaml` | Flash partitioning constraints + 3 typical layouts | ebuild `EosConfigValidator` | eboot build |
| `image.schema.yaml` | 128-byte image header format + signing specs | ebuild `EosBootIntegrator` | eboot `image_verify.c` |

## Integration flow

```
ebuild analyze → generates boot.yaml
                      ↓
ebuild generate-boot boot.yaml → generates:
  ├── eboot_flash_layout.h      (C header with #define constants)
  ├── eboot_memory.ld           (linker script memory regions)
  ├── eboot_config.cmake        (CMake definitions)
  └── pack_image.sh             (image packing + signing script)
                      ↓
eboot build consumes generated headers
                      ↓
cmake -B build -DEBLDR_BOARD=stm32f4 -C eboot_config.cmake
```

## Using generate_config.py

```bash
# Generate C headers from boot.yaml (eboot-side, no LLM)
python scripts/generate_config.py boot.yaml generated/

# Output:
#   generated/eboot_generated_layout.h
#   generated/eboot_generated_memory.ld
```

## Schema details

### boot.schema.yaml
- Flash layout: stage0 (≤16KB), stage1 (≤64KB), bootctl ×2, slot_a, slot_b
- Policy: max_boot_attempts, watchdog_timeout, require_signature, anti_rollback
- Image: hash algo (sha256/sha512), sign algo (ed25519/rsa2048/ecdsa_p256)
- RTOS: type, MPU mode, tick rate, priorities

### flash_layout.schema.yaml
- Constraints: stage0 ≤16KB, alignment 4KB, no overlap
- Typical layouts for 256KB, 1MB, 2MB flash

### image.schema.yaml
- 128-byte header: magic, version, size, SHA-256 hash, signature, flags
- Verification flow: magic → hash → signature → anti-rollback → accept
